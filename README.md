# VlorQl

VlorQl is a Rust-native, policy-driven **query orchestration framework**
for building safe LLM-driven data-query systems.

It implements the *Plan-Then-Execute* separation described in
[*Design Patterns for Securing LLM Agents against Prompt Injections*][paper]:
the LLM only emits a strictly typed `QueryPlan` JSON object.
VlorQl validates, authorizes, parameterizes, and compiles the plan
into SQL that can be safely sent to a downstream database engine.

The LLM never sees SQL and never has a direct path to the database.
Every user value is bound as a parameter, every column is enforced
against an allowlist or denylist, and every SQL feature is gated by
an explicit dialect profile.

[paper]: https://arxiv.org/abs/2504.13129

## Highlights

* **Plan-Then-Execute** — LLM output is a tagged `QueryPlan` JSON;
  SQL is never generated by the model.
* **Strict structured outputs** — `QueryPlan` is described by a
  `schemars`-derived JSON Schema, used both for prompt instructions
  and for LLM-side constraints (strict JSON Schema on OpenAI,
  Anthropic, GLM-4.7+, vLLM; `format: <schema>` on Ollama).
* **Multi-layer validation** — schema, policy (column/row), operand
  type checking, and dialect capability checks run before any SQL is
  built. Errors are aggregated and returned together so a single
  retry can fix every problem.
* **Controlled feature set** — `DialectProfile` lets operators
  disable CTEs, OFFSET, functions, join types, DISTINCT, GROUP BY
  column counts, and so on.
* **Always parameterized** — every literal value is bound as a
  placeholder (`$1`, `?`, or `LIMIT offset, limit` for MySQL). The
  compiler refuses to interpolate user input; an integration test
  suite guards against SQL-injection regressions.
* **Advanced SQL support** — the `QueryPlan` AST supports `COUNT(*)`,
  scalar subqueries, `WHERE expr IN (SELECT ...)`, and `EXISTS (SELECT ...)`,
  all compiled to dialect-correct SQL.
* **Query optimizer** — optional constant folding, predicate pushdown,
  column pruning, and join reordering between validation and compilation
  can reduce query cost by 30%+ on multi-table plans.
* **Built-in caching** — three-layer pluggable cache (SchemaCache,
  CompileCache, PromptCache) with TTL and LRU eviction, configurable
  from the `VlorQlBuilder`.
* **Multi-provider LLM clients** — `vlorql-llm` ships clients for
  OpenAI, Anthropic Claude, DeepSeek, Zhipu GLM, vLLM, and Ollama
  behind a single `LlmClient` trait. `create_llm_client`
  dispatches to the right implementation from an `LlmConfig`.
* **Streaming API** — `VlorQl::query_stream` yields raw text deltas
  as the LLM generates them, followed by a single
  `PlanComplete` event with the validated plan.
* **CLI** — `vlorql-cli` ships with `query` and `validate`
  subcommands that load TOML/JSON configs.

## Workspace layout

```text
crates/
├── vlorql-core/    # schema, errors, validation, compilation, prompt construction
├── vlorql-llm/     # LLM trait, multi-provider clients, mock client
├── vlorql/         # facade that ties everything together
├── vlorql-cli/     # command-line interface
└── vlorql/tests/   # integration + security tests
```

## Quick start

Add the crates to `Cargo.toml`:

```toml
[dependencies]
vlorql = { path = "crates/vlorql" }
vlorql-llm = { path = "crates/vlorql-llm" }
vlorql-core = { path = "crates/vlorql-core" }
tokio = { version = "1", features = ["full"] }
serde_json = "1"
```

Build a facade and run a question:

```rust
use std::sync::Arc;
use vlorql::{LlmConfig, LlmProvider, VlorQl};
use vlorql_core::policy::PolicyConfig;
use vlorql_core::schema::{
    ColumnSchema, DataType, SchemaMetadata, SchemaSnapshot, TableSchema,
};
use vlorql_llm::create_llm_client;

let schema = Arc::new(SchemaSnapshot::new(
    vec![TableSchema {
        name: "users".to_owned(),
        columns: vec![
            ColumnSchema {
                name: "id".to_owned(),
                data_type: DataType::Int,
                nullable: false,
                description: Some("User identifier".to_owned()),
                is_primary_key: true,
                foreign_key: None,
            },
            ColumnSchema {
                name: "name".to_owned(),
                data_type: DataType::String,
                nullable: false,
                description: Some("Display name".to_owned()),
                is_primary_key: false,
                foreign_key: None,
            },
        ],
        description: Some("Application users".to_owned()),
        primary_key: Some(vec!["id".to_owned()]),
    }],
    SchemaMetadata::default(),
));

let client = create_llm_client(LlmConfig {
    provider: LlmProvider::OpenAi,
    api_key: Some(std::env::var("OPENAI_API_KEY")?),
    model: "gpt-4o-mini".to_owned(),
    ..LlmConfig::default()
})?;

let vlorql = VlorQl::builder()
    .with_schema(schema)
    .with_dialect_name("postgres")
    .with_policy(PolicyConfig::default())
    .with_llm_client(client)
    .build()?;

let compiled = vlorql.query("Show user ids").await?;
println!("SQL:     {}", compiled.sql);
println!("Params:  {:?}", compiled.parameters);
```

### Run the runnable examples

```bash
# OpenAI / Anthropic / DeepSeek / Zhipu (set the relevant *_API_KEY env var)
DEEPSEEK_API_KEY=sk-... \
cargo run -p vlorql-llm --example multi_provider -- "List user ids"

# vLLM
vllm serve Qwen/Qwen2.5-7B-Instruct --port 8000 --guided-decoding-backend xgrammar
cargo run -p vlorql-llm --example local_vllm -- "List user ids"

# Ollama
ollama serve
ollama pull llama3.2
cargo run -p vlorql-llm --example local_ollama -- "List user ids"

# End-to-end pipeline (uses a mock LLM)
cargo run -p vlorql --example basic_query
```

## Highlights in detail

### Plan-Then-Execute

The LLM is asked to emit a single JSON object that matches the
`QueryPlan` schema. VlorQl never asks the model to write SQL,
never feeds SQL back to the model, and never lets the model see
the database driver. Every literal value is bound as a parameter;
every identifier goes through `QueryBuilder::quote_identifier` so
even attacker-controlled table/column names cannot break out of
their quoted form.

### Strict structured outputs

Each LLM client enables the strictest output mode the provider
supports:

| Provider   | Output mode                                                                   |
|------------|--------------------------------------------------------------------------------|
| OpenAI     | `response_format: { type: "json_schema", json_schema: { strict: true, ... } }` |
| Anthropic  | `output_config: { format: { type: "json_schema", schema: ... } }`             |
| DeepSeek   | `response_format: { type: "json_object" }` (DeepSeek does not support strict mode) |
| Zhipu      | `response_format: { type: "json_schema", ... }` on GLM-4.7+; `json_object` on older |
| vLLM       | `response_format.json_schema` (xgrammar / guidance / outlines / lm-format-enforcer) with a one-shot `json_object` fallback |
| Ollama     | `format: <JSON Schema object>` (with `format: "json"` fallback)                |

### Multi-layer validation

`VlorQl::validate_only` runs four stages in order and aggregates
every violation into a single `ValidationErrors`:

1. **Schema** — every base table and column reference must exist.
2. **Policy** — table allow/deny, column allow/deny, global denylist,
   mandatory row filters.
3. **Operand** — expression type checking (`5 = '5'` fails, `LIKE`
   on a numeric column fails, …).
4. **Dialect** — joins, CTEs, OFFSET, functions, JOIN types.

A single retry prompt the LLM sees contains every violation with
the offending value and a machine-actionable suggestion.

### Streaming queries

`VlorQl::query_stream` returns a
`Stream<Item = Result<StreamEvent, VlorQLError>>` that yields
`TextChunk` values as the LLM emits them, followed by a final
`PlanComplete` event (or `Error` if parsing or validation fails
after the LLM closes the connection).

```rust
use futures::StreamExt;
use vlorql::StreamEvent;

let mut stream = vlorql.query_stream("List user ids").await?;
let mut combined = String::new();
while let Some(item) = stream.next().await {
    match item? {
        StreamEvent::TextChunk(chunk) => {
            combined.push_str(&chunk);
            print!("[chunk] {chunk}");
        }
        StreamEvent::PlanComplete(plan) => println!("\nplan = {plan:?}"),
        StreamEvent::Error(error) => return Err(error.into()),
    }
}
```

## Testing

Run the full test suite:

```bash
cargo test --workspace
```

The integration tests in `crates/vlorql/tests/integration.rs` cover
the end-to-end pipeline with mock LLM clients. The security tests
in `crates/vlorql/tests/security/` cover SQL-injection,
policy-bypass, dialect-bypass, and malformed-JSON hardening. The
`vlorql-llm` integration tests in
`crates/vlorql-llm/tests/integration/multi_provider.rs` exercise
every supported LLM provider against a `mockito`-backed HTTP server.
