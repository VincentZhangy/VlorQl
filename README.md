VlorQl is a **Rust-native, policy-driven query orchestration framework** for building safe
LLM-powered data-query systems. It implements the **Plan-Then-Execute** separation
described in [*Design Patterns for Securing LLM Agents against Prompt Injections*][paper]:
the LLM emits a strictly typed `QueryPlan` JSON — **never SQL** — and VlorQl validates,
authorizes, parameterizes, and compiles it into safe SQL.

```
   User Question        ┌─ VlorQl ─────────────────────────────────────┐     SQL
   "Show all users" ──▶ │  LLM  ──▶ QueryPlan ──▶ Validate ──▶ Compile │──▶  DB
                        └──────────────────────────────────────────────┘
                              ▲                  ▲              ▲
                         DialectProfile     PolicyConfig    RewriteRules
```

The LLM never sees SQL, never touches the database driver, and every user value
is bound as a parameter. Column access is enforced by an allowlist/denylist,
and every SQL feature is gated by an explicit dialect profile.

[paper]: https://arxiv.org/abs/2504.13129

## Highlights

- **🔒 Plan-Then-Execute** — LLM emits typed `QueryPlan` JSON; SQL is never generated by the model.
  The model has zero direct access to the database.
- **📐 Strict structured outputs** — `QueryPlan` is described by a `schemars`-derived JSON Schema,
  enforced via provider-specific strict modes (OpenAI `strict: true`, Anthropic `json_schema`,
  GLM-4.7+, vLLM xgrammar, Ollama `format: <schema>`).
- **🛡️ Multi-layer validation** — Schema → Policy → Operand type-checking → Dialect capability gates.
  All errors are aggregated and returned together so a single retry fixes everything.
- **⚙️ Controlled feature set** — `DialectProfile` lets you disable CTEs, OFFSET, specific functions,
  join types, `DISTINCT`, window functions, and more.
- **🔐 Always parameterized** — every literal value is bound as a placeholder (`$1`, `?`).
  The compiler refuses to interpolate user input. SQL-injection regression tests guard every release.
- **🧠 Advanced SQL support** — `COUNT(*)`, scalar subqueries, `WHERE x IN (SELECT ...)`,
  `CASE WHEN`, `EXISTS`, window functions, `WITH RECURSIVE` CTEs, all compiled to dialect-correct SQL.
- **⚡ Query optimizer** — optional constant folding, predicate pushdown, column pruning,
  and join reordering. Reduces query cost by 30%+ on multi-table plans.
- **📦 Three-layer caching** — SchemaCache, CompileCache, PromptCache with TTL and LRU eviction.
- **🌐 Multi-provider** — ships clients for OpenAI, Anthropic Claude, DeepSeek, Zhipu GLM,
  vLLM, and Ollama behind a single `LlmClient` trait.
- **🎯 CLI** — `vlorql-cli` with `query` and `validate` subcommands, TOML/JSON config.

## Quick Start

### 1. Add dependency

```toml
[dependencies]
vlorql = { path = "crates/vlorql" }
vlorql-llm = { path = "crates/vlorql-llm" }
vlorql-core = { path = "crates/vlorql-core" }
tokio = { version = "1", features = ["full"] }
serde_json = "1"
```

### 2. Build and query

```rust
use std::sync::Arc;
use vlorql::{LlmConfig, LlmProvider, VlorQl};
use vlorql_core::policy::PolicyConfig;
use vlorql_core::schema::{
    ColumnSchema, DataType, SchemaMetadata, SchemaSnapshot, TableSchema,
};
use vlorql_llm::create_llm_client;

let schema = Arc::new(SchemaSnapshot::new(
    vec![TableSchema {
        name: "users".to_owned(),
        columns: vec![
            ColumnSchema {
                name: "id".to_owned(),
                data_type: DataType::Int,
                nullable: false,
                description: Some("User identifier".to_owned()),
                is_primary_key: true,
                foreign_key: None,
            },
            ColumnSchema {
                name: "name".to_owned(),
                data_type: DataType::String,
                nullable: false,
                description: Some("Display name".to_owned()),
                is_primary_key: false,
                foreign_key: None,
            },
        ],
        description: Some("Application users".to_owned()),
        primary_key: Some(vec!["id".to_owned()]),
    }],
    SchemaMetadata::default(),
));

let client = create_llm_client(LlmConfig {
    provider: LlmProvider::OpenAi,
    api_key: Some(std::env::var("OPENAI_API_KEY")?),
    model: "gpt-4o-mini".to_owned(),
    ..LlmConfig::default()
})?;

let vlorql = VlorQl::builder()
    .with_schema(schema)
    .with_dialect_name("postgres")
    .with_policy(PolicyConfig::default())
    .with_llm_client(client)
    .build()?;

let compiled = vlorql.query("Show user ids").await?;
println!("SQL:     {}", compiled.sql);
println!("Params:  {:?}", compiled.parameters);
```

### 3. Run the examples

```bash
# OpenAI / Anthropic / DeepSeek / Zhipu (set the relevant *_API_KEY env var)
DEEPSEEK_API_KEY=sk-... \
cargo run -p vlorql-llm --example multi_provider -- "List user ids"

# vLLM
vllm serve Qwen/Qwen2.5-7B-Instruct --port 8000 --guided-decoding-backend xgrammar
cargo run -p vlorql-llm --example local_vllm -- "List user ids"

# Ollama
ollama pull qwen2.5:7b
cargo run -p vlorql-llm --example local_ollama -- "List user ids"

# End-to-end pipeline (uses a mock LLM, no API key needed)
cargo run -p vlorql --example basic_query
```

## Configuration

VlorQl uses a **TOML or JSON** configuration file (default `vlorql.toml`, overridable via
`-c` / `VLORQL_CONFIG`). The file defines your database schema, dialect, security policy,
and LLM provider.

### Main config: `vlorql.toml`

```toml
# ── Schema (required) ──
[[schema.tables]]
name = "users"

[[schema.tables.columns]]
name = "id"
data_type = "Int"
nullable = false
primary_key = true

[[schema.tables.columns]]
name = "email"
data_type = "String"
nullable = true

# Foreign key
[[schema.tables.columns]]
name = "user_id"
data_type = "Int"
nullable = false
[ schema.tables.columns.foreign_key ]
foreign_table = "users"
foreign_column = "id"

# ── Dialect (optional) ──
[dialect]
dialect = "Postgres"              # Generic | Postgres | Mysql | Sqlite | BigQuery
quote_style = "DoubleQuoted"      # DoubleQuoted | Backtick | Bracket
supports_cte = true
supports_window_functions = true
max_joins = 20
allowed_join_types = ["inner", "left", "right", "full", "cross"]
denied_functions = ["pg_sleep", "pg_read_file"]
max_group_by_columns = 10
allow_distinct = true
supports_offset = true
supports_fetch = true

# ── Security Policy (optional) ──
[policy]
denied_columns = ["ssn", "password"]

[policy.table_policies.users]
denied_columns = ["email"]
row_filter = { column = "deleted_at", op = "is_null" }

# ── LLM (optional, CLI flags override these) ──
[llm]
provider = "openai"               # openai | anthropic | deepseek | zhipu | ollama | vllm
model = "gpt-4o"
api_base = "https://api.openai.com/v1"
api_key_env = "OPENAI_API_KEY"
```

### CLI usage

```bash
# Query with default vlorql.toml
cargo run -p vlorql-cli -- query -q "显示所有用户"

# Query with custom config
cargo run -p vlorql-cli -- query \
  -c my-config.toml \
  -q "按金额区间给订单打标" \
  --provider ollama \
  --model qwen2.5:7b \
  --api-base http://localhost:11434 \
  --dialect Postgres

# Validate a pre-existing plan file
cargo run -p vlorql-cli -- validate \
  -c vlorql.toml \
  --plan-file plan.json

# Environment variables
export VLORQL_CONFIG=./prod-config.toml
export OPENAI_API_KEY=sk-...
cargo run -p vlorql-cli -- query -q "查询订单总额"
```

### DialectProfile reference

| Field                     | Type                | Default         | Description                                            |
|---------------------------|---------------------|-----------------|--------------------------------------------------------|
| `dialect`                 | `SqlDialect`        | `Generic`       | SQL dialect (Postgres, Mysql, Sqlite, BigQuery)        |
| `quote_style`             | `IdentifierQuoting` | `DoubleQuoted`  | Identifier quoting style                               |
| `supports_cte`            | `bool`              | `true`          | Allow `WITH` CTEs                                      |
| `supports_window_functions` | `bool`            | `true`          | Allow window functions                                 |
| `supports_json_operations` | `bool`             | `true`          | Allow JSON operators (`->`, `->>`)                     |
| `max_joins`               | `Option<usize>`     | `None`          | Maximum join count                                     |
| `allowed_join_types`      | `Vec<JoinType>`     | `[]` (all)      | Allowed join types (inner, left, right, full, cross)   |
| `allowed_functions`       | `Vec<String>`       | `[]` (all)      | Allowed SQL functions (empty = allow all)              |
| `denied_functions`        | `Vec<String>`       | `[]`            | Denied SQL functions                                   |
| `max_group_by_columns`    | `Option<usize>`     | `None`          | Maximum GROUP BY columns                               |
| `allow_distinct`          | `bool`              | `true`          | Allow `COUNT(DISTINCT x)`                              |
| `allow_select_distinct`   | `bool`              | `true`          | Allow `SELECT DISTINCT`                                |
| `supports_offset`         | `bool`              | `true`          | Allow `OFFSET n`                                       |
| `supports_fetch`          | `bool`              | `true`          | Allow `FETCH FIRST n ROWS`                             |

### Prompt Skill: `small-model-skill.toml`

For small / local LLMs, you can define a **skill** that simplifies the output schema
and constrains query complexity:

```toml
name = "small-model"
description = "Simplifies output schema for small / local LLMs"
instructions = [
  "Keep queries simple: max 2 joins, no subqueries in WHERE.",
  "Always use table aliases (first letter of table name).",
  "Prefer LEFT JOIN over NOT IN / NOT EXISTS.",
  "GROUP BY must include every non-aggregated column in SELECT.",
]
simplify_schema = true
forbid_features = ["set_operation", "window_functions", "ctes", "distinct_on"]
disable_output_fields = ["set_operation", "ctes", "distinct_on"]
```

### SQL Rewrite Rules: `rewrite-rules.toml`

Post-process the compiled SQL with regex rules:

```toml
[[rules]]
name = "strip-final-semicolons"
match_pattern = ";\\s*$"
replace_template = ""
description = "Remove trailing semicolons from generated SQL"

[[rules]]
name = "pg-math-precision"
match_pattern = "(?i)\\bINTEGER\\s*:\\s*(\\d+)\\s*/\\s*(\\d+)"
replace_template = "NUMERIC($1) / NUMERIC($2)"
description = "Cast integer division to numeric for PostgreSQL precision"
```

## Validation Pipeline

`VlorQl::validate_only` runs four stages and aggregates all violations:

1. **Schema** — every base table and column reference must exist.
2. **Policy** — table/column allow/deny, global denylist, mandatory row filters.
3. **Operand** — expression type checking (`5 = '5'` fails, `LIKE` on a numeric column fails).
4. **Dialect** — joins, CTEs, OFFSET, functions, JOIN types checked against the profile.

A single retry prompt contains every violation with the offending value and a
machine-actionable suggestion.

## Streaming API

```rust
use futures::StreamExt;
use vlorql::StreamEvent;

let mut stream = vlorql.query_stream("List user ids").await?;
let mut combined = String::new();
while let Some(item) = stream.next().await {
    match item? {
        StreamEvent::TextChunk(chunk) => {
            combined.push_str(&chunk);
            print!("[chunk] {chunk}");
        }
        StreamEvent::PlanComplete(plan) => println!("\nplan = {plan:?}"),
        StreamEvent::Error(error) => return Err(error.into()),
    }
}
```

## Testing

```bash
# Full test suite
cargo test --workspace

# Core only
cargo test -p vlorql-core

# LLM clients only
cargo test -p vlorql-llm

# Security tests (SQL injection, policy bypass, malformed JSON)
cargo test -p vlorql --test security
```

The integration tests in `crates/vlorql/tests/integration.rs` cover the end-to-end
pipeline with mock LLM clients. Security tests in `crates/vlorql/tests/security/`
cover SQL-injection, policy-bypass, dialect-bypass, and malformed-JSON hardening.
